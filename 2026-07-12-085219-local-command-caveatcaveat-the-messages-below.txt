     TyLog: Android vault-sync stability & usability fixes

     Context

     Full-repo audit of TyLog (Flutter journal app, Obsidian-style .typ vault,
     Nextcloud/WebDAV sync, SAF file access on Android). The sync protocol
     itself is sound — If-Match etag guards, conflict snapshots, atomic SAF
     writes with backup/rollback, mass-delete guard. The real Android usage
     problems are performance/battery, a data-loss window on app kill, error
     noise while typing, and a couple of non-atomic desktop writes. This plan
     fixes those with the smallest diffs that work.

     Findings (ranked)

     1. Every sync re-hashes the whole vault over SAF.
     lib/nextcloud_sync.dart:216 calls storage.hash(path) for every local file
     on every run; on Android each hash is a method-channel call + full file
     read (SafBridge.kt:394-405), plus separate exists/stat SAF round-trips per
     path (lines 212, 215). With the 25 s poll + 10 s autosave sync, a large
     vault means constant full re-reads → slow syncs, battery drain, UI jank.
     This is the #1 Android usage problem.
     2. Up to ~700 ms of typing lost when Android kills the app.
     didChangeAppLifecycleState (lib/app_mobile.dart:1290-1299) only handles
     resumed; autosave is a 700 ms debounce (app_mobile.dart:670). Nothing
     flushes on paused/hidden.
     3. Typing during sync surfaces as a sync failure.
     _requireLocalReplacementAllowed (nextcloud_sync.dart:739-743) throws
     StateError('Local edit started during sync...'), which the catch in
     _syncNow (app_mobile.dart:1151-1161) shows as an error status. Self-heals
     but looks broken to the user, repeatedly.
     4. Desktop local writes are not atomic. LocalVaultStorage.writeBytes
     (lib/vault_storage.dart:120-127) does write-tmp → delete target → rename
     (crash window where the file is gone; POSIX rename already replaces
     atomically) and uses a fixed .tmp name (concurrent-write collision). Same
     delete-then-rename in nextcloud_sync.dart:133-137;
     vault_registry.dart:293-301 writes vaults.json with no temp file at all.
     (Android's SafBridge.writeAtomic is already correct.)
     5. Trace log write amplification. _appendTrace
     (nextcloud_sync.dart:956-972) reads + rewrites the entire
     .tylog/sync_trace.jsonl as a full atomic SAF write, 2+ times per sync.
     6. Remote mass-delete guard is all-or-nothing. nextcloud_sync.dart:196-200
     only refuses when local listing is completely empty. A flaky
     DocumentsProvider that drops a subset (listing + per-path exists both go
     through the same SAF resolve) could propagate spurious deletions.

     Deliberately skipped (say so, don't build):
     - Background sync (WorkManager/foreground service) — big dependency +
     battery/permission complexity; resume-triggered sync covers the local-first
     model. Add only if users report stale multi-device state as a real pain.
     - File watchers, crash-reporting SaaS, vault schema migration — YAGNI for
     now.

     Changes

     1. Hash fast-path + reuse listing (fixes finding 1) — 
     lib/nextcloud_sync.dart

     - Add localSize to SyncCursor (nextcloud_sync.dart:1224-1250,
     fromJson/toJson; old state files lack it → field null → falls back to
     hashing once, then self-populates).
     - Change _localFiles usage: keep the VaultStorageEntry map from
     storage.list(recursive: true) (entries already carry size/modified) instead
     of collapsing to a path set, and use it for localExists/localStat in the
     loop — eliminates the per-path exists() + stat() SAF round-trips at lines
     212, 215.
     - In the loop (line 216): if prev?.localSha256 != null && prev.localMillis 
     == stat.modified.millis && prev.localSize == stat.size → reuse
     prev.localSha256, skip storage.hash. Null mtime/size → hash as today.
     - Persist localSize wherever cursors are written after
     upload/download/skip.
     - Effect: steady-state sync goes from O(vault bytes) reads to one list()
     call + hashing only changed files.

     2. Flush on background (fixes finding 2) — lib/app_mobile.dart:1290

     In didChangeAppLifecycleState, add: on paused (and hidden), if dirty, call
     unawaited(_save(syncAfter: false)). ~4 lines.

     3. Treat edit-during-sync as benign re-queue (fixes finding 3)

     Replace the generic StateError in _requireLocalReplacementAllowed
     (nextcloud_sync.dart:739-743) with a dedicated exception type (e.g. class 
     SyncDeferred implements Exception); in _syncNow's catch
     (app_mobile.dart:1151), on SyncDeferred just _queueCloudSync() and set a
     neutral status ("Sync deferred while editing") instead of syncError.

     4. Atomic desktop writes (fixes finding 4)

     - LocalVaultStorage.writeBytes (vault_storage.dart:120-127): unique temp
     name (.tmp-<pid or timestamp>), drop the delete(), rename directly over
     target.
     - Same pattern for nextcloud.json (nextcloud_sync.dart:133-137) and
     vaults.json (vault_registry.dart:293-301) — ideally route both through one
     small writeFileAtomic helper (or reuse LocalVaultStorage.writeBytes where a
     storage is at hand).

     5. Trace log: one write per sync, capped (fixes finding 5) — 
     nextcloud_sync.dart:948-972

     Buffer trace events in memory during sync(), write once in a finally, and
     trim to the last ~200 lines when rewriting. Drops SAF churn to one file
     write per sync.

     nextcloud_sync.dart:196-200

     deletions exceed max(10, 25% of syncState.length), throw the same kind of
     StateError ("refusing to propagate N deletions; local listing may be
     incomplete") so it surfaces in the sync dashboard like today's guard. //

⏺ All six fixes are implemented and verified: flutter analyze is clean and all
  117 tests pass, including three new ones.

  What was wrong and what changed:

  1. Every sync re-read and hashed the whole vault over SAF — twice per file
  (once to detect changes, once more when writing the cursor). This was the #1
  Android battery/speed problem, firing on the 25 s poll. Now SyncCursor stores
  localSize, and when a file's mtime+size match the cursor the stored hash is
  reused; the loop also uses the single recursive list() result (which already
  carries size/mtime from DocumentsContract) instead of per-file exists/stat
  channel calls. Steady-state sync does one SAF listing and zero file reads.
  2. Up to 700 ms of typing was lost if Android killed the app inside the
  autosave debounce. didChangeAppLifecycleState now flushes dirty edits on
  paused/hidden.
  3. Typing during a sync showed a red sync error. That abort is now a dedicated
  SyncDeferred exception; the app shows "Sync deferred while editing" and
  quietly re-queues instead of surfacing an error.
  4. Desktop writes could leave a file missing after a crash (write-tmp → delete
  target → rename, with a shared .tmp name). Replaced with one writeFileAtomic
  helper (unique temp + direct atomic rename) used by LocalVaultStorage,
  nextcloud.json, and vaults.json. The Android SAF path was already correct.
  5. The sync trace log was fully read and rewritten 2+ times per sync. Events
  are now buffered and written once per sync in a finally (a hard process kill
  loses that run's trace — marked with a ponytail: comment as an accepted
  ceiling).
  6. The mass-delete guard only fired on a totally empty listing. Added a
  proportional guard: more than max(10, 25 % of tracked files) disappearing
  locally in one run aborts the sync instead of deleting remote files —
  protection against a flaky DocumentsProvider dropping listing entries.

  New tests: second sync performs zero hash calls; mass local disappearance
  throws and leaves the remote intact; concurrent overwrites are atomic with no
  temp leftovers. One existing test was updated to expect SyncDeferred instead
  of StateError — that's the intended behavior change from fix 3.

  Worth checking by hand on a device (I can't drive Android here): open a large
  vault and watch the second sync complete near-instantly; background the app
  mid-typing, kill it from recents, reopen — the last keystrokes should be
  there.

  Skipped, deliberately: background sync via WorkManager — add only if stale
  multi-device state becomes a real complaint; resume-triggered sync covers the
  local-first design.
