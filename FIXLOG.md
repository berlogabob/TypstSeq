## 2026-07-16 — can't type/newline after an inserted task (rich editor)
- Root cause: the task chip is an atomic protected block; with the caret at the chip's **trailing edge** (mobile's natural spot, and where `loadSource` lands when a note ends with a task) `insertNewline` refused the edit and typing was mangled by `_replaceAcrossBlocks` → validation revert ("Edit kept safe…"). Device testing revealed the symmetric **leading edge** case too (tapping the chip line puts the caret before the chip).
- Fix (`lib/rich_editor.dart`): `insertNewline` now opens a writable empty paragraph after (trailing) or above (leading) the protected block; `_handleValue` gained a pure-insertion branch that routes typing at either chip edge through that paragraph. Edits ON the chip stay refused; backspace at the boundary still deletes the whole card (guard test green).
- 5 new tests incl. one replaying the exact on-device note; suite 222 pass / 1 skip; analyze clean. Device: attached-console run showed the real-IME Enter at the boundary handled with no error (previously reverting).
- Follow-ups spotted while verifying: (1) toggling a task from the **Agenda** while its note is open in the editor writes to disk but the open editor keeps a stale copy — next editor save can undo the toggle (pre-existing editor/disk coherence gap); (2) the "Edit kept safe" banner wording is scary for what is a refused edit.

## 2026-07-16 — sync stuck on unresolvable journal conflicts (device-verified fix)
- Live device forensics: THREE stacked conflict records for `daily/2026/07/2026-07-16.typ`, all with **byte-identical local/remote snapshots** — spurious. Cause A: the concurrent-edit check filed a conflict when the app's own 400 ms autosave wrote the editor's text to disk mid-sync (`diskSource != sourceBeforeSync` without checking `diskSource == source`). Cause B (independent, code-traced): sync-created conflicts freeze `remoteEtag` at record time, the sync loop never refreshes skipped unresolved paths, so `resolveConflict`'s "Nextcloud changed again" guard throws forever once the remote moves — a hard in-app deadlock.
- Fixes: (1) no conflict when disk == current editor content; (2) conflicts replace, never stack, per path (`_discardConflictsForPath`); (3) `loadSyncConflicts` self-heals records whose snapshots are byte-identical (deletes them); (4) the `unresolved-conflict` sync branch refreshes the record's remote snapshot + etag when the remote moved, so resolution succeeds after the next sync (guard kept for the sync→dialog window).
- 5 new tests (incl. the previously untested conflict-with-etag + remote-moved + resolve combination); suite 241 pass / 1 skip; analyze clean. **Device: on first launch of the fixed build the 3 stuck records self-healed to zero with no user action, and sync attempts resumed immediately** (first run failed only on `Failed host lookup` — phone offline at that moment; recovers with network).
